# AND-24 Cross-Repo Panex Portable Runner

## 1. Problem Frame

**Signal:** AND-23 production dogfood proved that Панэкс can answer a digest question on Fly.io and then handle natural Russian follow-ups through exact source_expand without source_key jargon. The next product-value problem is choosing what improves real day-to-day use from other repos now: portability, continuity, decision usefulness, long-run ergonomics, or external evidence depth, without breaking explicit-only behavior or turning Панэкс into broad web search.

**Constraints:**
- Панэкс remains explicit-only; no automatic calls from Codex/Claude by default.
- Default external links remain author-supplied references unless a future mode is explicitly requested.
- Do not add a hidden full UI Reduce/Meta/Reddit pipeline to agent-facing calls.
- Keep default answers compact, source-backed, and practitioner-opinion framed.
- Preserve exact source handles and source_expand audit path.
- Respect Fly.io latency/cost constraints and long-running nature of all-expert calls.
- Do not require the user to know API jargon such as source_key/source_expand.

**Acceptance:** Produce a ranked set of next upgrade variants after AND-23 with clear product value, risk, implementation effort, and a recommended next slice. The recommendation should preserve explicit-only invocation, production Fly.io target, source provenance, compact answers, and cross-repo usability.

## 2. Decision

**Selected:** AND-24 Cross-Repo Panex Portable Runner

**Selection policy:** Choose the next slice that maximizes day-to-day cross-repo use while preserving explicit-only invocation, production Fly.io target, compact source-backed answers, and low pipeline creep.

**Why selected:** This is the highest immediate product-value slice after AND-23 because Панэкс is meant to be used from other repos. A global panex runner removes cwd/localhost confusion without adding backend state, hidden LLM calls, automatic search, or external-link fetching.


**Invariants:**
- Панэкс remains explicit-only; no automatic calls from parent agents.
- Production Fly.io is the default target for real panex ask/expand calls.
- Localhost is only used through explicit --local or --api-url debug paths.
- AGENT_CONTEXT_API_TOKEN is required for live calls and must never be printed or embedded in the installed shim.
- expert_digest remains the default response mode; source_bundle is explicit raw/audit mode only.
- source_expand remains exact handle lookup, not a new search or digest run.

**Pre-conditions:**
- [ ] Experts Panel production Agent Context API is deployed on Fly.io.
- [ ] A valid production AGENT_CONTEXT_API_TOKEN is available in the calling environment/backend .env.
- [ ] The local Experts_panel checkout has a working backend virtualenv for the shim target.

**Post-conditions:**
- [ ] A global panex command can be installed and called from a foreign cwd.
- [ ] panex ask defaults to Fly.io expert_digest and ignores ambient local API URLs.
- [ ] panex expand defaults to Fly.io source_expand and ignores ambient local API URLs.
- [ ] panex doctor reports setup status without printing secrets.
- [ ] Repo-local and user-level Панэкс agent instructions use panex as the real-call interface.

## 3. Rationale

**Counterargument:** A result-id/snapshot layer would improve follow-up continuity more, but it adds state, expiry, privacy, and storage decisions before the basic cross-repo invocation substrate is stable.

**Selected variant weakest link:** The runner must be production-safe without leaking tokens, silently falling back to localhost, or becoming another confusing CLI surface.

**Rejected alternatives:**
| Variant | Verdict | Reason |
|---------|---------|--------|
| AND-24 Cross-Repo Panex Portable Runner | **Selected** | This is the highest immediate product-value slice after A... |
| Digest Snapshot And Result IDs | Rejected | Higher continuity value but requires state/expiry/privacy design; better after portable invocation is stable. |
| Consensus And Dissent Map | Rejected | Useful for decisions but changes answer synthesis behavior; less foundational than making current Панэкс usable from any repo. |
| Expert Roster And Capability Manifest | Rejected | Low risk and useful, but less directly fixes the current invocation friction than a portable runner. |

**Evidence requirements:**
- Local BDD/TDD tests for panex defaults, explicit local debug, source_bundle opt-in, token safety, doctor, and install shim.
- Contract tests proving Claude/Codex agent instructions use panex ask/expand and source_bundle opt-in.
- Production dogfood from a foreign cwd against Fly.io for ask and expand.
- Documentation updated in agent-context spec, documentation map, and dogfood report.

**Predictions:**
| Claim | Observable | Threshold |
|-------|------------|-----------|
| Foreign-cwd panex ask will call production Fly.io expert_digest even if AGENT_CONTEXT_API_URL points to localhost. | BDD test captures panex.requests.post URL and payload. | URL equals https://experts-panel.fly.dev/api/v1/agent/context and payload response_mode is expert_digest. |
| Foreign-cwd panex expand will call production Fly.io source_expand even if AGENT_CONTEXT_EXPAND_API_URL points to localhost. | BDD test captures panex.requests.post URL and payload. | URL equals https://experts-panel.fly.dev/api/v1/agent/context/expand and payload contains exact source_keys. |
| Production dogfood from /private/tmp will work end to end. | panex ask and panex expand commands against Fly.io. | ask returns mode=expert_digest with warnings=[]; expand returns mode=source_expand with not_found=[] and warnings=[]. |

## 4. Consequences

**Rollback plan:**
Triggers:
- panex defaults to localhost or ambient local API URL for real calls
- token appears in stdout/stderr/docs/shim
- production ask/expand fails due runner URL/path setup
- agent instructions become ambiguous enough to call localhost for real requests
Steps:
1. Revert panex CLI and install script changes.
2. Restore agent instructions to explicit low-level --api-url production calls.
3. Remove AND-24 docs/status claims until fixed.
Blast radius: User-level panex command, repo-local/user-level agent instructions, docs, CLI tests; no backend API runtime change.

**Refresh triggers:**
- Panex runner is packaged as MCP/skill/plugin instead of shim.
- Production Agent Context URL changes.
- Agent Context token model changes.
- source_bundle or source_expand defaults change.

**Affected files:** backend/src/cli/panex.py, scripts/install_panex_runner.sh, backend/tests/test_agent_context_cli.py, backend/tests/test_experts_panel_researcher_contract.py, backend/tests/test_experts_panel_researcher_dogfood.py, .codex/agents/experts_panel_researcher.toml, .claude/agents/experts_panel_researcher.md, docs/architecture/agent-context-api.md, docs/DOCUMENTATION_MAP.md, docs/quality/panex-portable-runner-dogfood-2026-05-07.md
