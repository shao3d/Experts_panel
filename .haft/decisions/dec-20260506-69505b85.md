# V1 Evidence Quality Labels

## 1. Problem Frame

**Signal:** Experts Panel now has a production explicit-only agent-facing API with expert_digest and source_expand. The next question is which upgrade increases practical value for Codex/Claude Code users without turning the API into automatic web search, a second digest pipeline, or an overbuilt platform.

**Constraints:**
- Keep explicit-only invocation; no automatic Панэкс calls for generic tasks.
- Keep practitioner posts framed as source-backed opinions, not proven facts.
- Avoid reintroducing the full UI Reduce/Meta/Comment synthesis pipeline by default.
- Keep Панэкс user language human-readable; users should not need API jargon such as source_key/source_expand.
- Respect Fly.io production constraints: long all-expert requests are natural, but default UX should not require repeated expensive runs.
- Do not fetch/crawl external links unless a future mode is explicitly requested.

**Acceptance:** Produce a ranked set of next upgrade variants with clear product value, implementation cost/risk, and recommended next slice.

## 2. Decision

**Selected:** V1 Evidence Quality Labels

**Selection policy:** Maximize trust-calibrated day-to-day usefulness for other repos while keeping implementation, latency, and pipeline-creep risk low. High pipeline-creep options are deferred unless they are explicit opt-in modes.

**Why selected:** Evidence Quality Labels improves the immediate usefulness of both existing agent-facing endpoints by helping Панэкс explain whether practitioner evidence is deep, shallow, announcement-level, indirect, comment-supported, or noisy. It raises trust calibration without adding a new broad pipeline, external crawling, or long-running platform mechanics.


**Invariants:**
- Do not present practitioner opinions as proven facts.
- Do not add default external-link crawling or fetching.
- Do not reintroduce the full UI Reduce/Meta/Reddit pipeline for the agent-facing API by default.
- Do not add secrets to docs, agent prompts, tests, or repo files.
- Keep human Панэкс language in Russian; source_key/API jargon may be hidden behind agent instructions.
- Preserve exact source references so the user can expand evidence when needed.

**Pre-conditions:**
- [ ] Current agent-facing API has /api/v1/agent/context with expert_digest/source_bundle behavior.
- [ ] Current agent-facing API has /api/v1/agent/context/expand for source-level evidence expansion.
- [ ] Панэкс remains explicit-only: no automatic Experts Panel calls from Codex/Claude by default.
- [ ] Agent-facing search keeps the existing production Fly.io target and authentication model.

**Post-conditions:**
- [ ] Agent-facing digest and/or expansion responses expose lightweight evidence-quality calibration for important sources.
- [ ] Labels distinguish at minimum deep/practical signal, announcement/mention-level signal, indirect/weak signal, and comment-supported versus comment-noise cases.
- [ ] Панэкс instructions explain how to phrase these labels naturally in Russian without calling them proof.
- [ ] BDD/TDD tests cover the new calibration behavior on fixtures and contract-level outputs.
- [ ] Documentation describes the labels, their limits, and how they relate to /context and /context/expand.

**Admissibility:**
- NOT: No new mandatory LLM call solely to label every source unless explicitly justified and measured.
- NOT: No default external link enrichment.
- NOT: No hidden automatic Панэкс invocation.
- NOT: No 'доказано' framing for human opinions.
- NOT: No broad platform/cache rewrite in the same slice.

## 3. Rationale

**Counterargument:** Selector-Based Expansion may feel more directly ergonomic because the user can ask 'раскрой источники по Рефату' without source_key mechanics, and async/cache may become important for slow all-expert runs.

**Selected variant weakest link:** Labels can become subjective or overconfident unless they are tied to observable source features and phrased as calibration rather than proof.

**Rejected alternatives:**
| Variant | Verdict | Reason |
|---------|---------|--------|
| V1 Evidence Quality Labels | **Selected** | Evidence Quality Labels improves the immediate usefulness... |
| V2 Selector-Based Expansion | Rejected | High UX value, but it needs request snapshots or selector semantics and is more stateful; better after evidence quality vocabulary is stable. |
| V3 Explicit External Link Enrichment Mode | Rejected | Potentially valuable later, but too much latency/security/cost/pipeline-creep risk for the next slice. |
| V4 Async Jobs And Result Cache | Rejected | Addresses long-running operations but not the immediate answer-quality/trust problem; larger platform step. |
| V5 Answer Profiles For Workflows | Rejected | Could be mostly prompt/template work, but risks format sprawl and over-synthesis before evidence calibration is solid. |
| V6 Expert Roster And Capability Manifest Endpoint | Rejected | Low-risk portability improvement, but lower immediate impact on answer quality than V1. |

**Evidence requirements:**
- Contract/acceptance tests show labels are present and source-linked where expected.
- Tests show shallow announcement-like sources are not described as strong evidence.
- Tests show comments can be marked as useful support or mostly noise without dropping raw expansion capability.
- Production dogfood with 2-3 experts confirms labels help interpret returned sources without excessive latency growth.
- Docs and Панэкс agent instructions are updated together.

**Predictions:**
| Claim | Observable | Threshold |
|-------|------------|-----------|
| Evidence quality labels will make Панэкс answers easier to trust and triage in other repos. | In dogfood output, strong/deep sources and shallow/announcement-level sources are visibly distinguishable without expanding every raw source. | At least 2 production examples show useful distinction between strong and weak/shallow sources. |
| The slice will not materially increase endpoint latency if implemented as lightweight calibration over already selected sources. | Production smoke for 2-3 experts completes in the same expected minutes-scale range as current expert_digest without an additional broad reduce pass. | No systematic extra full-pipeline or all-source LLM pass is introduced. |
| Evidence expansion remains raw-enough and does not become another digest layer. | /api/v1/agent/context/expand still returns source-linked raw/detail fields while adding only lean calibration metadata or notes. | Expanded response preserves source handles and raw comments/external link metadata behavior. |

## 4. Consequences

**Rollback plan:**
Triggers:
- Labels are misleading or overconfident in dogfood.
- Latency/cost increases because implementation adds a new broad LLM pass.
- Agent answers start hiding raw evidence behind another digest-like abstraction.
- Users find the labels confusing or too API-jargony.
Steps:
1. Disable label rendering in Панэкс instructions first.
2. If backend fields were added, keep schema backward-compatible and stop populating optional labels.
3. Revert documentation claims about evidence-quality labels until behavior is corrected.
Blast radius: Agent-facing API responses and Панэкс presentation only; core UI pipeline and stored expert data should remain unaffected.

**Refresh triggers:**
- After AND-20 implementation dogfood.
- After adding selector-based expansion or external link enrichment.
- If source schema or digest structure changes.
- If Fly.io latency/cost constraints change.

**Affected files:** docs/architecture/agent-context-api.md, backend/src/services/agent_context_service.py, backend/src/api/agent_context_endpoint.py, backend/src/cli/agent_context.py, backend/src/cli/agent_context_expand.py, .codex/agents/experts_panel_researcher.toml, .claude/agents/experts_panel_researcher.md, backend/tests/test_agent_context_api.py, backend/tests/test_agent_context_acceptance.py, backend/tests/test_agent_context_cli.py, backend/tests/test_experts_panel_researcher_contract.py
