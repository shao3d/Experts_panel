# AND-23 Selector-Based Expansion UX

## 1. Problem Frame

**Signal:** AND-22 production dogfood showed Панэкс is already useful as an explicit-only research helper: compact answers, weak-signal honesty, human follow-up source expansion, external-link boundary, and exact scope discipline all passed. The next problem is choosing upgrades that increase day-to-day product value in other repos without turning Панэкс into automatic web search, an overbuilt platform, or a hidden second synthesis pipeline.

**Constraints:**
- Панэкс remains explicit-only; no automatic calls from Codex/Claude by default.
- Default external links stay references-only unless a future mode is explicitly requested.
- Do not add a broad always-on pipeline or hidden full UI Reduce/Meta synthesis.
- Keep default answers compact and source-backed.
- Preserve source handles/source_expand as the audit path.
- Respect Fly.io latency/cost constraints for long expert runs.

**Acceptance:** Produce a ranked set of next upgrade variants with clear product value, implementation risk, and recommended next slice. The recommendation should preserve explicit-only use, source provenance, compact answers, and production Fly.io practicality.

## 2. Decision

**Selected:** AND-23 Selector-Based Expansion UX

**Selection policy:** Maximize day-to-day Панэкс value and follow-up ergonomics while preserving explicit-only invocation, compact answers, source provenance, low pipeline creep, and low Fly.io latency/cost risk.

**Why selected:** AND-22 showed the digest -> human follow-up -> source_expand workflow is the highest-value current Панэкс loop. Selector-based expansion improves day-to-day follow-up ergonomics with low implementation effort, no backend API change, no new retrieval, no external fetching, and no extra production latency beyond the existing exact source_expand lookup.


**Invariants:**
- Панэкс remains explicit-only; no automatic expert calls.
- No backend API change for AND-23; reuse existing expert_digest and source_expand.
- Expansion phrases must map to previous digest source handles only.
- No source_expand when previous digest/source handles are missing or the selector is ambiguous.
- Do not expand all sources unless the user explicitly asks for all/raw/concrete handles.
- source_expand final output stays a lean Evidence Note, not a second digest/reduce/synthesis.

**Pre-conditions:**
- [ ] AND-19 source_expand by source_key exists and is deployed.
- [ ] AND-22 product-quality evaluator and dogfood scenarios exist.
- [ ] Repo-local Claude/Codex Панэкс instructions are tracked.

**Post-conditions:**
- [ ] Repo-local and user-level Панэкс instructions support natural Russian selector phrases.
- [ ] Product-quality scenarios cover named expert, claim, comments, weak-source, ambiguous selector, and no-previous-digest boundaries.
- [ ] Local BDD tests pass.
- [ ] Production dogfood demonstrates at least selected AND-23 behaviors on Fly.io.

**Admissibility:**
- NOT: Do not add a new backend endpoint or result_id store in this slice.
- NOT: Do not add external link fetching/enrichment.
- NOT: Do not add hidden new expert_digest/source_bundle calls for expansion phrases.
- NOT: Do not rely on raw memory to invent source_key handles.

## 3. Rationale

**Counterargument:** Digest Snapshot And Result IDs would make follow-up selection more reliable across context loss and should be considered next if selector dogfood remains brittle.

**Selected variant weakest link:** Context-only selector mapping can still be brittle if the previous digest is absent from the parent/subagent context or contains many similar claims.

**Rejected alternatives:**
| Variant | Verdict | Reason |
|---------|---------|--------|
| AND-23 Selector-Based Expansion UX | **Selected** | AND-22 showed the digest -> human follow-up -> source_exp... |
| Digest Snapshot And Result IDs | Rejected | Higher long-term reliability, but requires state ownership, expiry, privacy, and API/storage policy. Defer until context-only selector UX proves insufficient. |
| Answer Profiles For Workflows | Rejected | Cheaper packaging improvement, but less direct leverage on the already-proven source expansion workflow. |
| Explicit External Link Enrichment Mode | Rejected | Valuable later, but higher security/provenance/latency risk than improving selected evidence expansion first. |

**Evidence requirements:**
- Local pytest for product evaluator and agent contract passes.
- Production dogfood answers show source_expand for selected handles and clarification/no-call behavior for ambiguous or missing-context cases.
- Docs/spec map reflect AND-23 behavior and boundaries.

**Predictions:**
| Claim | Observable | Threshold |
|-------|------------|-----------|
| AND-23 can be implemented with no backend API changes. | git diff touches agent instructions, evaluator/tests, docs/Haft, but not agent_context endpoint/service runtime. | No backend runtime API files changed. |
| Selector UX improves follow-up without requiring source_key jargon. | BDD scenarios and production dogfood include Russian human phrases such as 'раскрой по Рефату' and 'что там в комментариях'. | All AND-23 local scenarios pass and production dogfood has no source_key-jargon requirement in user prompts. |
| Ambiguity/missing previous digest boundaries prevent accidental API calls. | Evaluator scenarios require clarification/boundary answers without Request passport/source_keys_sent. | Clarification/boundary scenarios pass and bad guessing/rerun samples fail. |

## 4. Consequences

**Rollback plan:**
Triggers:
- Панэкс guesses handles when previous digest context is missing.
- Панэкс expands too many sources by default.
- Production dogfood shows repeated selector ambiguity or wrong expert/source choice.
Steps:
1. Revert AND-23 agent instruction changes.
2. Remove or relax AND-23 evaluator scenarios if they encode incorrect product behavior.
3. Reopen the problem and choose Digest Snapshot And Result IDs.
Blast radius: Repo-local/user-level agent instructions and product-quality guardrail tests; backend API remains unaffected.

**Refresh triggers:**
- After 5-10 real user Панэкс follow-up sessions.
- If parent/subagent context loss causes wrong selector expansion.
- Before implementing result_id/snapshot state.

**Affected files:** .codex/agents/experts_panel_researcher.toml, .claude/agents/experts_panel_researcher.md, backend/tests/fixtures/panex_quality_scenarios.json, backend/scripts/panex_quality_eval.py, backend/tests/test_panex_quality_eval.py, backend/tests/test_experts_panel_researcher_contract.py, docs/architecture/agent-context-api.md, docs/quality/panex-product-quality-rubric.md, docs/quality/panex-selector-expansion-dogfood-2026-05-07.md, docs/DOCUMENTATION_MAP.md
