# Solutions for: Next Panex product-value upgrades after AND-22 dogfood

Problem: prob-20260507-e9fbae6a

## Variants (8)

### V1. Selector-Based Expansion UX

Let users ask natural follow-ups such as 'раскрой по Рефату', 'покажи слабые места', or 'дай источники по этому тезису' and have Панэкс reliably choose the right digest handles from the previous answer, with optional small backend/CLI support if current context-only selection proves brittle.

**Strengths:**
- Directly improves the workflow AND-22 proved valuable
- Low pipeline risk if it remains source_expand-based
- Makes Панэкс feel more natural in real chats

**Novelty:** Improves the second-step evidence workflow rather than the first digest answer.

**Weakest link:** Requires reliable mapping from human phrases to previous digest handles; brittle if previous context is missing or too long.

**Risks:**
- May need a lightweight snapshot/id mechanism later
- Could over-expand if selection rules are vague

**Stepping stone:** yes — opens future possibilities

**Stepping-stone basis:** Sets up richer evidence explorer and selector commands later without changing retrieval.

### V2. Answer Profiles For Workflows

Add explicit answer styles such as 'быстро', 'decision memo', 'compare table', 'implementation checklist', and 'risk review' that Панэкс can use when the user asks for a specific working mode.

**Strengths:**
- Cheap to implement mostly in agent instructions/evaluator
- Useful across repos and tasks
- No backend latency

**Novelty:** Optimizes final answer packaging, not retrieval or evidence storage.

**Weakest link:** Can become prompt/template sprawl and may over-format answers instead of improving substance.

**Risks:**
- Too many modes can confuse users
- Could regress compactness

### V3. Digest Snapshot And Result IDs

Persist or locally cache a compact digest snapshot with stable result_id so follow-up expansion can reference 'last answer', 'second source', or 'Akimov's strongest source' even across parent/subagent context loss.

**Strengths:**
- Solves a real follow-up reliability problem
- Makes source_expand robust across chats/agents
- Enables future UI/CLI evidence explorer

**Novelty:** Adds continuity/state to the current stateless digest -> expand workflow.

**Weakest link:** Introduces state ownership, expiry, privacy, and storage policy decisions.

**Risks:**
- More backend/API complexity
- Needs cleanup/expiry and auth boundaries

**Stepping stone:** yes — opens future possibilities

**Stepping-stone basis:** Foundation for selector expansion, async jobs, and result cache.

### V4. Expert Roster And Capability Manifest

Expose or package a small machine-readable manifest of expert ids, UI names, aliases, groups, domains, and what each expert is usually good for so Панэкс can select experts more confidently in other repos.

**Strengths:**
- Very low risk
- Improves cross-repo setup and expert selection
- Reduces ambiguity and clarification loops

**Novelty:** Improves discoverability and portability rather than answer synthesis.

**Weakest link:** Value depends on keeping metadata accurate as roster evolves.

**Risks:**
- Lower immediate answer-quality uplift
- Needs docs discipline

**Stepping stone:** yes — opens future possibilities

**Stepping-stone basis:** Helps future automatic expert suggestions while preserving explicit-only behavior.

### V5. Explicit External Link Enrichment Mode

Add an opt-in command to fetch and summarize selected author-supplied external links from a digest/source_expand result with clear provenance, fetch_status, timestamp, and security limits.

**Strengths:**
- High user value when experts link GitHub/docs/articles
- Closes a visible gap from external_links dogfood
- Can remain explicit and selected-link-only

**Novelty:** Turns references-only links into verified external context only when explicitly requested.

**Weakest link:** Security/provenance/copyright/latency risks are much higher than current references-only mode.

**Risks:**
- Network/security complexity
- Can drift into web research product
- May be slow or noisy

### V6. Async Jobs And Result Cache

Add async job submission/status/result retrieval for long all-expert or heavy production calls, so Codex/Claude can start a Панэкс query and return later without blocking the chat for many minutes.

**Strengths:**
- Useful for all-experts and heavy queries
- Reduces blocking pain
- Pairs well with result IDs

**Novelty:** Optimizes operational usability for long-running calls rather than evidence quality.

**Weakest link:** Requires job lifecycle, storage, auth, expiry, and error handling on Fly.io.

**Risks:**
- Platform complexity
- Not needed for common 2-4 expert calls
- More deploy/ops surface

**Stepping stone:** yes — opens future possibilities

**Stepping-stone basis:** Pairs naturally with Digest Snapshot And Result IDs.

### V7. Panex Quality Gallery And Regression Pack

Turn high-quality dogfood answers into a small golden gallery with expected behavior notes, edge cases, and periodic regression prompts to keep product quality stable over time.

**Strengths:**
- Low risk
- Makes future regressions visible
- Helps new chats/agents understand good Панэкс behavior

**Novelty:** Improves product QA and onboarding rather than user-facing API capability.

**Weakest link:** Indirect product value; can become documentation work if not kept lean.

**Risks:**
- Does not add a new capability
- Requires periodic maintenance

### V8. Consensus And Dissent Map

Offer an explicit mode that turns selected expert digests into a compact map of agreement, disagreement, minority views, and confidence/weakness by source handles.

**Strengths:**
- Highly aligned with 'opinion of experts' product value
- Useful for PM/architecture decisions
- Can be answer-level first without backend changes

**Novelty:** Improves the analytical value of expert opinions, not just evidence access.

**Weakest link:** May require another synthesis layer and can overstate agreement if sources are weak.

**Risks:**
- Risk of over-synthesis
- Needs careful proof-avoidance framing

**Stepping stone:** yes — opens future possibilities

**Stepping-stone basis:** Can start as an explicit answer profile, then become structured output if useful.
