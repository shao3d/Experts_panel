# Solutions for: Expose Experts Panel as an agent-facing research/context API

Problem: prob-20260504-0d81ecb4

## Variants (5)

### V1. Thin authenticated REST context endpoint

Add a narrow read-only endpoint such as `POST /api/v1/agent/context` that wraps the existing query pipeline into a stable non-UI contract: request intent, expert filters, recency, include_reddit, response mode, and a structured response with answer/context, sources, confidence, request_id, timings, and abstention/error states.

**Strengths:**
- Fastest path to a real usable MVP.
- Small blast radius: can coexist with current `/api/v1/query`.
- Easy to test with FastAPI/TestClient and curl.
- Good stepping stone for OpenAPI docs and later MCP wrapper.

**Novelty:** Turns the existing UI/SSE query workflow into a small agent-facing JSON contract without changing the core pipeline or adopting a new protocol first.

**Weakest link:** If the contract is too close to current UI response shape, agents may get prose instead of reusable context and future MCP wrapping may require a second redesign.

**Risks:**
- May become a one-off REST API if output schema is not designed as a reusable context bundle.
- Requires auth/rate/cost controls before any real external exposure.
- Non-streaming wrapper may hide long-running progress unless response modes are explicit.

**Stepping stone:** yes — opens future possibilities

**Stepping-stone basis:** A stable REST schema can later be exposed as an MCP tool/resource or consumed by internal agent scripts.

**Rollback:** Disable/remove the new router without touching existing `/api/v1/query`; keep data model unchanged.

### V2. Remote MCP server over Streamable HTTP

Expose Experts Panel directly as an MCP server with tools such as `research_context`, `compare_tools`, `trend_scan`, and resources/prompts for source schemas. Use Streamable HTTP and authentication for remote agents.

**Strengths:**
- Best fit for MCP-aware agents and tool registries.
- Can expose tools/resources/prompts in a discoverable way.
- Aligns with current agent ecosystem direction for external context providers.

**Novelty:** Makes the system agent-native via MCP discovery and tool invocation rather than just another HTTP API.

**Weakest link:** Protocol/auth/security complexity can dominate the first slice and distract from validating whether agents need this specific research context.

**Risks:**
- Remote MCP requires careful authentication, Origin validation, and tool-permission design.
- Harder to debug and test than a thin REST endpoint.
- May overfit to MCP before the product contract is proven.

**Stepping stone:** yes — opens future possibilities

**Stepping-stone basis:** A later MCP adapter can wrap the same internal context service used by REST.

**Rollback:** Keep MCP server as a separate adapter layer over an internal service; disable adapter without removing the core context service.

### V3. Private local agent bridge first

Create a local/private bridge for Codex or internal agents only: repo-local script/MCP-like tool that calls the existing backend or direct service functions and returns structured context. No public endpoint yet.

**Strengths:**
- Lowest external security/cost exposure.
- Good for dogfooding prompts, schema, and source bundle design.
- Can run before production deployment work.

**Novelty:** Validates agent workflows privately before opening any network-facing surface.

**Weakest link:** It may prove local utility while avoiding the actual hard problem: stable external contract, auth, deployment, and third-party integration.

**Risks:**
- May produce false confidence because only the user's own workflow is tested.
- External consumers still need a later API/protocol layer.
- Local bridge can drift from production endpoint behavior.

**Stepping stone:** yes — opens future possibilities

**Stepping-stone basis:** Can refine the context bundle schema before public API work.

**Rollback:** Delete local bridge files; no production exposure.

### V4. Full AI Architect workflow API

Build the larger product direction now: an API for uploading/accepting business tasks, decomposing them, running parallel expert/community retrieval, and returning a full strategy document with evidence and recommendations.

**Strengths:**
- Closest to the long-term strategic vision in `AI Architect Mode`.
- Higher perceived product value than a raw context API.
- Can produce complete deliverables for business users.

**Novelty:** Targets the high-value AI consulting product rather than a low-level context endpoint.

**Weakest link:** Too large for first validation; product, UX, data, retrieval, and synthesis risks are all coupled in one step.

**Risks:**
- High scope and slower feedback.
- Harder to isolate failures: decomposition, retrieval, synthesis, evidence, or UX.
- May delay the simpler agent-facing context layer that other agents could reuse.

**Rollback:** Difficult to roll back cleanly if many workflow-specific services/schema are added.

### V5. Source bundle retrieval API without final synthesis

Expose a read-only context retrieval endpoint that returns ranked source bundles, snippets, metadata, confidence, and suggested follow-up queries, leaving final reasoning/synthesis to the calling agent.

**Strengths:**
- Lower LLM cost than full answer synthesis.
- Very agent-friendly for callers that want to control final reasoning.
- Can make source traceability very strong.

**Novelty:** Positions Experts Panel as a context provider instead of an answer generator.

**Weakest link:** Users may expect direct analytical answers; external agents may misuse or over-trust partial snippets without the system's synthesis/validation layer.

**Risks:**
- Less immediately impressive to human users.
- Requires careful schema design to prevent context dumping.
- May underuse existing reduce/meta-synthesis strengths.

**Stepping stone:** yes — opens future possibilities

**Stepping-stone basis:** Can be one `response_mode` under V1 rather than a separate product path.

**Rollback:** Keep as an internal mode; remove external route if not useful.

## Dominated Variant Elimination

- Full AI Architect workflow API — dominated by Thin authenticated REST context endpoint, Source bundle retrieval API without final synthesis. For the first MVP, full AI Architect has worse effort, security/cost exposure, learning speed, and future optionality than a narrow context API plus source-bundle mode. Keep it as a later product layer, not the first endpoint.

## Pareto Front Trade-Offs

- Thin authenticated REST context endpoint — Best balanced first MVP: fast learning, moderate agent fit, good evidence traceability, future MCP optionality. Weakness is that it must avoid becoming UI-prose-in-JSON.
- Source bundle retrieval API without final synthesis — Best context-provider shape for agents and strongest traceability/cost profile, but likely less satisfying alone. Best implemented as a response mode under V1.
- Remote MCP server over Streamable HTTP — Best native agent fit, but security/protocol complexity makes it better as adapter step after a stable internal contract exists.
- Private local agent bridge first — Lowest risk dogfooding path; useful as a preflight/probe, but insufficient as the external endpoint strategy.

**Selection policy:** Pick the smallest reversible first slice that proves external-agent utility while preserving source traceability, auth/rate/cost controls, and existing UI/SSE behavior. Prefer a stepping-stone contract that can later be wrapped as MCP or expanded into AI Architect.

**Recommendation (advisory):** Thin authenticated REST context endpoint

**Recommendation rationale:** V1 is the best first implementation recommendation because it creates a stable external contract quickly, reuses the existing pipeline, is easy to test, and keeps future MCP optionality. V5 should be folded into V1 as a `response_mode=source_bundle` rather than treated as a separate product. V2 is likely the second step once the internal response schema is proven. V4 is strategically attractive but too broad for first validation.
