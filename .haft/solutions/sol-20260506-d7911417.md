# Solutions for: Next product-value upgrades for Experts Panel agent-facing API

Problem: prob-20260506-09a7d0ff

## Variants (6)

### V1. Evidence Quality Labels

Add lightweight source-quality calibration to Панэкс/API output: distinguish deep argument vs announcement/mention, comments signal vs noise, weak/indirect/stale evidence, and when a source merely supports vs changes the digest. Prefer agent-side/structured-field light layer before any new LLM pipeline.

**Strengths:**
- Directly addresses production dogfood finding where akimov:1972 was relevant but shallow/announcement-level.
- Helps parent agents decide when to ask for more evidence or discount a source.
- Low latency if implemented as deterministic/agent-side labels first.

**Novelty:** Improves trust and calibration of existing digest/expand results rather than adding new retrieval scope.

**Weakest link:** May become subjective unless labels are tightly tied to observable source features and not presented as proof.

**Risks:**
- Could drift into another mini-digest if overworded.
- Needs careful BDD examples for source depth/comment noise.

**Stepping stone:** yes — opens future possibilities

**Stepping-stone basis:** Creates vocabulary for later evaluation, ranking, and source-selector improvements.

**Rollback:** Remove labels/instructions without changing retrieval endpoints.

### V2. Selector-Based Expansion

Extend source expansion so Панэкс can send human selectors such as expert_id + top_n, key_signal/supporting_sources, or request_id + expert selector, instead of always materializing source_keys in the agent. This makes 'покажи источники по Рефату' robust even when source handles are not visible in the parent chat.

**Strengths:**
- Directly improves human UX and reduces source_key leakage.
- Makes follow-up expansion more reliable across agents/repositories.
- Can keep source_expand as the same endpoint with selector fields instead of a third endpoint.

**Novelty:** Moves some follow-up source selection from fragile agent context into an explicit API contract.

**Weakest link:** Requires storing or passing the previous digest/request snapshot safely; stateless production API design becomes more complex.

**Risks:**
- State/request_id storage can introduce security, expiry, and cache invalidation issues.
- Could blur boundary between expansion and new query if selectors are too broad.

**Stepping stone:** yes — opens future possibilities

**Stepping-stone basis:** Foundation for saved research sessions and follow-up workflows.

**Rollback:** Keep old source_keys path and disable selector fields.

### V3. Explicit External Link Enrichment Mode

Add an opt-in mode to fetch/summarize author-supplied external links from selected sources, with strict caps, fetch_status, provenance, and security rules. Trigger only on phrases like 'проверь ссылки' or 'обогати внешние ссылки'.

**Strengths:**
- Many expert posts contain GitHub, docs, papers, articles that may be central evidence.
- Could greatly increase answer usefulness for tools/libs/approaches.
- Leverages existing external_links metadata.

**Novelty:** Adds a new evidence type beyond Telegram posts/comments while preserving explicit-only behavior.

**Weakest link:** High security/latency/cost risk and can turn Панэкс into general web browsing if boundaries are weak.

**Risks:**
- Network failures, paywalls, dynamic pages, cloning temptation, hallucinated summaries.
- Needs URL allow/deny policy and strong provenance.

**Rollback:** Keep metadata-only external links; disable enrichment mode.

### V4. Async Jobs And Result Cache

Introduce request/job handles for long expert_digest runs, cache repeated production results, and let Панэкс poll/reuse recent results instead of making users wait minutes for 3+ experts or all-expert calls.

**Strengths:**
- Directly addresses long production digest latency.
- Supports all-experts workflows more comfortably.
- Could reduce cost for repeated dogfood queries.

**Novelty:** Improves operational UX and Fly.io fit rather than evidence quality itself.

**Weakest link:** Adds platform complexity: persistence, TTL, auth isolation, cache invalidation, and job lifecycle UI/CLI behavior.

**Risks:**
- Bigger implementation than a narrow feature slice.
- Could complicate CLI/subagent contract and error modes.

**Stepping stone:** yes — opens future possibilities

**Stepping-stone basis:** Foundation for persistent research sessions and selector-based expansion by request_id.

**Rollback:** Keep synchronous endpoint as canonical; make jobs optional.

### V5. Answer Profiles For Workflows

Let Панэкс format the same expert_digest into explicit workflow profiles such as 'architecture decision', 'tool selection', 'risk review', or 'market scan', without changing retrieval. Mostly subagent contract/templates, possibly request metadata later.

**Strengths:**
- High perceived value in other repos where user wants decision-ready help.
- Low backend cost if agent-side only.
- Can be explicit-only and reversible.

**Novelty:** Improves downstream usability by shaping the answer to the task, not by changing evidence retrieval.

**Weakest link:** Can become prompt/template sprawl and may over-synthesize practitioner opinions into decision claims.

**Risks:**
- May duplicate current Signals frame and confuse output expectations.
- Harder to test objectively.

**Rollback:** Remove profiles from agent instructions; keep API unchanged.

### V6. Expert Roster And Capability Manifest Endpoint

Expose a small authenticated/read-only manifest for agent clients: available experts, groups, display names, aliases, supported modes, and current endpoint capabilities. This reduces hardcoded agent config drift across repos.

**Strengths:**
- Solves repo-local vs user-level config drift for expert names/groups.
- Good for future external agent integrations.
- Low risk and easy to test.

**Novelty:** Improves portability and correctness of Панэкс in other repositories without touching search quality.

**Weakest link:** Lower direct user-visible value than evidence improvements; can become yet another config surface if not kept small.

**Risks:**
- Needs clear source of truth for roster; frontend/backend config duplication risk.
- Does not improve answers by itself.

**Stepping stone:** yes — opens future possibilities

**Stepping-stone basis:** Foundation for portable clients and multi-repo Панэкс setup checks.

**Rollback:** Keep static agent config aliases.

## Non-Dominated Set

**Computed Pareto front:** Evidence Quality Labels, Selector-Based Expansion, Expert Roster And Capability Manifest Endpoint

## Dominated Variant Elimination

- Explicit External Link Enrichment Mode — dominated by Evidence Quality Labels. Explicit external-link enrichment can add value later, but for the next slice V1 gives comparable day-to-day value, better trust calibration, and much lower effort, latency, security, and pipeline-creep risk.
- Async Jobs And Result Cache — dominated by Evidence Quality Labels. Async jobs/cache address long-running operations, but they do not directly improve evidence quality; V1 gives equal practical value for the current product question with lower effort and better reversibility.
- Answer Profiles For Workflows — dominated by Evidence Quality Labels. Answer profiles are cheaper than backend work, but they risk prompt/template sprawl and less trustworthy over-formatting; V1 improves trust and answer quality more directly.

## Pareto Front Trade-Offs

- Evidence Quality Labels — Best next slice for trust and practical answer quality with low effort and low latency risk; it does not solve stateless follow-up selection by itself.
- Selector-Based Expansion — Best human follow-up UX for commands like 'раскрой источники по Рефату', but it needs request snapshots or selector semantics and has more backend/schema complexity.
- Expert Roster And Capability Manifest Endpoint — Very low-risk portability and discoverability improvement for cross-repo Панэкс use, but lower immediate answer-quality value than V1.

**Selection policy:** Maximize trust-calibrated day-to-day usefulness for other repos while keeping implementation, latency, and pipeline-creep risk low. High pipeline-creep options are deferred unless they are explicit opt-in modes.

**Recommendation (advisory):** Evidence Quality Labels

**Recommendation rationale:** V1 improves the immediate usefulness of both existing agent-facing endpoints by helping the user understand whether returned practitioner evidence is deep, shallow, announcement-level, indirect, comment-supported, or noisy. It does not require another broad pipeline, external crawling, or long-running platform mechanics, and it is a stepping stone for better expansion and source selection later.
