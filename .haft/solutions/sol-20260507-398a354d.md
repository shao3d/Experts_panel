# Solutions for: Next Panex product-value upgrades after AND-23 selector expansion

Problem: prob-20260507-9b9b89ff

## Variants (9)

### V1. Cross-Repo Panex Portable Runner

Make Панэкс reliably callable from any repo by adding a repo-independent wrapper/runner contract: global/user-level CLI command or small installed script, env/token preflight, production Fly URL pinning, and docs/tests proving the agent does not depend on the current Experts_panel cwd.

**Strengths:**
- Directly matches the intended use from other projects
- Low backend risk
- Can catch missing production token or wrong path before a live call
- Reduces fragile dependence on Experts_panel/backend .venv being the current repo

**Novelty:** Moves product value from repo-local capability to cross-repo daily usability.

**Weakest link:** Needs careful token/path setup so it is portable without leaking secrets or silently falling back to localhost.

**Risks:**
- May touch user-level setup outside repo
- Needs clear install/update story
- Could duplicate existing CLI if not scoped tightly

**Stepping stone:** yes — opens future possibilities

**Stepping-stone basis:** Creates the stable invocation substrate for result IDs, roster manifests, and future MCP/skill packaging.

### V2. Digest Snapshot And Result IDs

Persist or locally cache compact digest snapshots with stable result_id/source handle maps so follow-up expansion can reference 'last answer', 'second source', or a claim even when parent/subagent context is lost.

**Strengths:**
- Solves the known weakest link from AND-23
- Improves auditability and follow-up reliability
- Foundation for async jobs/result cache

**Novelty:** Adds continuity/state to the current stateless digest -> source_expand workflow.

**Weakest link:** Introduces storage ownership, expiry, privacy, auth, and cleanup policy decisions.

**Risks:**
- More backend/API complexity
- State can become stale or leak sensitive query context if not scoped
- Needs schema and expiry decisions

**Stepping stone:** yes — opens future possibilities

**Stepping-stone basis:** Foundation for async jobs, evidence explorer, and robust cross-chat follow-ups.

### V3. Expert Roster And Capability Manifest

Expose/package a machine-readable expert manifest with expert ids, UI names, aliases, groups, domains, and 'good for' hints so Панэкс can select experts and ask clarifications more confidently across repos.

**Strengths:**
- Low implementation risk
- Improves portability and user trust
- Reduces ambiguity around Russian/display names and groups

**Novelty:** Improves expert discoverability and selection before the query rather than answer synthesis after the query.

**Weakest link:** Metadata can rot unless tied to source-of-truth expert config and tested.

**Risks:**
- Lower immediate wow effect than new answer capabilities
- Needs docs discipline as roster evolves

**Stepping stone:** yes — opens future possibilities

**Stepping-stone basis:** Enables later expert recommendation mode while preserving explicit confirmation.

### V4. Consensus And Dissent Map

Add an explicit answer mode/profile that turns selected expert digests into a compact map of agreement, disagreement, minority views, confidence/weakness, and source handles.

**Strengths:**
- Highly aligned with 'what do experts think?' use case
- Useful for decisions in other repos
- Can start as answer-level behavior without backend runtime changes

**Novelty:** Improves PM/architecture decision usefulness, not evidence access or transport.

**Weakest link:** Can over-synthesize weak practitioner opinions into fake consensus if the prompt/rubric is not strict.

**Risks:**
- Risk of proof-like consensus framing
- May need BDD product scenarios to prevent overstatement
- Adds another presentation mode

**Stepping stone:** yes — opens future possibilities

**Stepping-stone basis:** Can later become a structured profile or report mode if dogfood shows value.

### V5. Answer Profiles For Workflows

Add explicit user-selectable answer profiles such as quick take, decision memo, implementation checklist, risk review, and compare table, while keeping default compact signals unchanged.

**Strengths:**
- Cheap and fast
- No backend latency
- Useful across repos when the user knows the desired output type

**Novelty:** Optimizes final answer packaging for different work modes rather than retrieval/storage.

**Weakest link:** Can become prompt/template sprawl and reduce substance if too many modes are added.

**Risks:**
- Mode confusion
- May regress compact defaults
- Can overlap with Consensus/Dissent Map

### V6. Async Jobs And Result Cache

Add async submission/status/result retrieval for long all-expert or heavy production calls so Панэкс can start a long job and return later instead of blocking the parent chat.

**Strengths:**
- Important for all-expert queries
- Reduces blocking pain
- Pairs naturally with result IDs

**Novelty:** Improves operational ergonomics for long-running production calls rather than answer quality.

**Weakest link:** Requires job lifecycle, persistence, auth, expiry, and Fly.io operational care.

**Risks:**
- Platform complexity
- Not needed for common 2-4 expert calls
- Can distract from core product UX

**Stepping stone:** yes — opens future possibilities

**Stepping-stone basis:** Depends on or benefits from result IDs/snapshots.

### V7. Explicit External Link Enrichment Mode

Add an opt-in mode to fetch and summarize selected author-supplied external links from digest/source_expand with provenance, fetch_status, timestamp, and safety limits.

**Strengths:**
- High value when experts link GitHub/docs/articles
- Closes visible gap from external_links dogfood
- Can stay selected-link-only

**Novelty:** Turns references-only links into verified external context only when explicitly requested.

**Weakest link:** Security, copyright, latency, and provenance risk are much higher than current references-only behavior.

**Risks:**
- Network/security complexity
- Can drift into broad web research
- Potential copyright and token budget issues

### V8. Panex Quality Gallery And Regression Pack

Turn high-quality dogfood answers into a small gallery of expected behavior, edge cases, anti-patterns, and periodic regression prompts to keep product quality stable over time.

**Strengths:**
- Low risk
- Helps future chats and agents
- Protects against regressions in tone/scope/proof framing

**Novelty:** Improves QA/onboarding and product memory rather than adding a user-facing capability.

**Weakest link:** Indirect product value; can become documentation work if not kept small and executable.

**Risks:**
- No direct new capability
- Maintenance burden if too large

### V9. Explicit Expert Recommendation Preflight

Let the user ask Панэкс to suggest which experts/group to query for a topic, using the manifest and current roster, then require confirmation before making the actual digest call.

**Strengths:**
- Useful when the user does not remember all expert names
- Reduces wrong-scope queries
- Pairs with roster manifest

**Novelty:** Adds a pre-query selection helper while preserving explicit confirmation before any expert API call.

**Weakest link:** Can feel like automation creep if it silently chooses experts or calls the API without confirmation.

**Risks:**
- Needs strong confirmation boundary
- May add one extra interaction turn
- Depends on manifest quality

**Stepping stone:** yes — opens future possibilities

**Stepping-stone basis:** Builds on Expert Roster And Capability Manifest.

## Dominated Variant Elimination

- Async Jobs And Result Cache — dominated by Digest Snapshot And Result IDs. Async jobs are useful for long all-expert runs, but snapshots/result IDs solve the continuity foundation first with lower platform complexity.
- Explicit External Link Enrichment Mode — dominated by Digest Snapshot And Result IDs, Consensus And Dissent Map. External enrichment can be valuable later, but it carries much higher security/provenance/latency risk than improving continuity or decision synthesis first.
- Panex Quality Gallery And Regression Pack — dominated by Expert Roster And Capability Manifest. Quality gallery helps QA, but a roster/manifest gives more direct cross-repo product value at similar low risk.

## Pareto Front Trade-Offs

- Cross-Repo Panex Portable Runner — Best immediate product-value and portability step; weakest on continuity because it does not preserve previous digest handles across context loss.
- Digest Snapshot And Result IDs — Best continuity/provenance step; higher implementation/security complexity because it introduces state.

- Answer Profiles For Workflows — Cheapest packaging improvement; risks template sprawl and lower structural value than V1/V2.
- Explicit Expert Recommendation Preflight — Good expert-selection UX if backed by a manifest and confirmation boundary; depends on V3 quality.
